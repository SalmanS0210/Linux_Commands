In the cloud, "Connectivity" is about how data gets in and out (Networking & SSH), while "Observability" is about knowing what’s happening once it's inside (Services & Logs).

1. Networking: The Digital Plumbing
In Linux, networking isn't just about "being online"—it’s about managing ports and interfaces.

Essential Diagnostic Tools
ip a: Shows your IP addresses and network interfaces. Look for eth0 or ens3 (your cloud network card).

ss -tulpn: The "Who is listening?" command. It shows which services are using which ports (e.g., Nginx on port 80).

ping: Checks basic connectivity to another server.

curl -I [URL]: Fetches the "headers" of a website to see if a web server is responding without downloading the whole page.

💡 Cloud Tip: If ping or curl fails to your instance, check your Cloud Security Group/Firewall settings first. Linux might be ready, but the cloud provider might be blocking the port.

2. SSH (Secure Shell): Your Remote Portal
Since your server is in the cloud, SSH is your only way in.

Key-Based Authentication
Passwords are "brute-forceable." Keys are not.

Generate a key pair on your local machine: ssh-keygen -t ed25519.

Public Key (.pub): Goes on the server in ~/.ssh/authorized_keys.

Private Key: Stays on your laptop. Never share this.

Hardening SSH (/etc/ssh/sshd_config)
For your final project, you'll need to edit this file to make your server safer:

PasswordAuthentication no: Disables passwords entirely (keys only).

PermitRootLogin no: Forces you to log in as a regular user first, then use sudo.

3. Services (systemd): The Engines
Almost everything in Linux (Nginx, Docker, Databases) runs as a service managed by systemd.

The systemctl Command
This is the only command you need to manage the "life" of a service:

Bash
sudo systemctl start nginx    # Start it now
sudo systemctl stop nginx     # Stop it now
sudo systemctl enable nginx   # Make it start automatically when the server boots
sudo systemctl status nginx   # Check if it's running or if it crashed
4. Logs: The Black Box Recorder
When a service fails to start, the logs will tell you why. Linux stores logs in /var/log.

How to Read Logs
tail -f /var/log/syslog: "Follows" the main system log in real-time. Great for watching errors happen as you type.

journalctl -u nginx: Shows logs specifically for the Nginx service.

journalctl -xe: Shows the last few lines of the system log with extra detail—perfect for when a service fails to start.

# Interview quetions 

1. Difference Between ping and curl
ping

Tests network connectivity using ICMP packets.

ping google.com

Checks:

DNS resolution
Network reachability
Packet loss
Latency

Example:

64 bytes from google.com

curl

Tests application-level connectivity (HTTP/HTTPS/API).

curl https://google.com

Checks:

DNS resolution
TCP connection
HTTP/HTTPS response
API responses

Example:

HTTP/1.1 200 OK
Interview Answer

ping checks whether a host is reachable over the network using ICMP, while curl checks whether an application or web service is responding over HTTP/HTTPS.

2. How Does SSH Key Authentication Work?

SSH uses a public-private key pair.

Generate Keys
ssh-keygen

Creates:

~/.ssh/id_rsa       (private key)
~/.ssh/id_rsa.pub   (public key)
Copy Public Key to Server
ssh-copy-id user@server

Public key stored in:

~/.ssh/authorized_keys
Authentication Process
Client requests SSH connection.
Server sends a challenge.
Client signs challenge using private key.
Server verifies using public key.
Access granted.
Advantages
More secure than passwords
Enables automation
Common in CI/CD pipelines
Interview Answer

SSH key authentication uses a public-private key pair. The server stores the public key, and the client proves its identity by signing a challenge with the private key without transmitting the private key itself.

3. Difference Between Restart and Reload in systemd
Restart
systemctl restart nginx

What happens:

Stops service completely
Starts it again

Effects:

Existing connections may be interrupted
New process is created
Reload
systemctl reload nginx

What happens:

Reloads configuration files
Service keeps running

Effects:

Existing connections remain active
No downtime (if supported)
Example

After modifying:

/etc/nginx/nginx.conf

Use:

nginx -t
systemctl reload nginx
Interview Answer

restart stops and starts the service, causing potential downtime. reload re-reads configuration without stopping the service, allowing configuration changes with minimal impact.

4. How Do You Find Which Process Is Using Port 8080?
Using ss
ss -tulpn | grep 8080

Example:

LISTEN 0 128 *:8080 users:(("java",pid=1234))
Using lsof
sudo lsof -i :8080

Output:

COMMAND PID USER
java    1234 app
Using netstat
netstat -tulpn | grep 8080
Kill Process
kill -9 1234
Interview Answer

I usually use ss -tulpn | grep 8080 or lsof -i :8080 to identify the process and PID listening on port 8080.

5. Where Would You Check Logs If Nginx Fails to Start?
Service Status

First place to check.

6. How Would You Troubleshoot a Website Returning 502/503 Errors?
What They Mean
502 Bad Gateway

Nginx/Load Balancer cannot communicate with backend application.

Example:

Nginx -> Application Server

Application is down.

503 Service Unavailable

Service unavailable due to:

Application stopped
Maintenance mode
Resource exhaustion
Troubleshooting Steps
Step 1: Check Nginx
systemctl status nginx
Step 2: Check Backend Application

Example:

systemctl status myapp

or

docker ps
Step 3: Check Listening Ports
ss -tulpn

Verify backend port exists:

8080
9000
5000
Step 4: Test Backend Directly
curl localhost:8080

Expected:

200 OK
Step 5: Check Logs

Nginx:

tail -f /var/log/nginx/error.log

Application:

journalctl -u myapp

or

docker logs container_name
Step 6: Check Resources

CPU:

top

Memory:

free -m

Disk:

df -h
Step 7: Verify Nginx Upstream Configuration

Example:

upstream app {
    server 127.0.0.1:8080;
}

Ensure backend is actually listening on that port.

Interview Answer

For 502/503 errors, I first check Nginx and backend application status, verify backend ports are listening, test the backend directly using curl, review Nginx and application logs, check system resources, and validate the upstream configuration. Most 502 errors occur when the backend service is down or unreachable.

Production Scenario Example

User reports:

https://app.company.com returns 502

Commands I'd run:

systemctl status nginx

ss -tulpn | grep 8080

curl localhost:8080

journalctl -u nginx -n 50

tail -f /var/log/nginx/error.log

df -h

free -m

This sequence quickly identifies whether the issue is with Nginx, the application, networking, or server resources.
